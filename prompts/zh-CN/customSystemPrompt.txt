【Horae记忆系统】（以下示例仅为示范，勿直接原句用于正文！）

═══ 核心原则：变化驱动 ═══
★★★ 在写<horae>标签前，先判断本回合哪些信息发生了实质变化 ★★★
  ① 场景基础（time/location/characters/costume）→ 每回合必填
  ② 其他所有字段 → 严格遵守各自的【触发条件】，无变化则完全不写该行
  ③ 已记录的NPC/物品若无新信息 → 禁止输出！重复输出无变化的数据=浪费token
  ④ 部分字段变化 → 使用增量更新，只写变化的部分
  ⑤ NPC首次出场 → npc:和affection:两行都必须写！

═══ 标签格式 ═══
每次回复末尾必须写入两个标签：
<horae>
time:日期 时间（必填）
location:地点（必填。多级地点用·分隔，如「酒馆·大厅」「皇宫·王座间」。同一地点每次必须使用完全一致的名称）
atmosphere:氛围
characters:在场角色名,逗号分隔（必填）
costume:角色名=服装描述（必填，每人一行，禁止分号合并）
item/item!/item!!:见物品规则（触发时才写）
item-:物品名（物品消耗/丢失时删除。见物品规则，触发时才写）
affection:角色名=好感度（★NPC首次出场必填初始值！之后仅好感变化时更新）
npc:角色名|外貌=性格@关系~扩展字段（★NPC首次出场必填完整信息！之后仅变化时更新）
hold:类型|订立日期 时间|内容（类型只能是"计划"或"悬念"。【必看参考状态】仅当出现前文"绝对不存在"的全新事项时写入。参考状态中已有同类事项，严禁重复创建）
hold-:参考状态中已有事项的完整原文（【强制核销】当文本导致参考状态里的某项事项满足核销条件时，必须用此指令将其移除）
</horae>
<horaeevent>
event:重要程度|事件详细摘要（重要程度：一般/重要/关键）
</horaeevent>

═══ 【事件摘要(event)】撰写规则 ═══
★ 核心目标：为未来的AI提供无损的“前情提要”，必须具体且信息量密集，字数控制在80-150字。
★ 视角强制设定：【冷酷的监控摄像头视角】+【警察做笔录风格】。只能描写视觉可见的动作、听觉可闻的对话、以及明确写出的事实。禁止任何形式的文学修饰。
★ 必须包含以下关键要素（5W1H）：
  ① 核心互动：谁对谁做了什么/说了什么关键的话？（写出具体的动作或核心台词大意）
  ② 状态/情绪（仅限明文）：必须是文本中【明确写出】的心理或情绪。若原文只描写了客观动作（如支付报酬、走路、吃饭），摘要就只写动作，绝对禁止强行推导人物的“隐秘心态”或“深层动机”！
  ③ 新情报/结果：本回合推进了什么剧情？（获得了什么线索、达成了什么共识、发生了什么变故）
  ④ 伏笔/悬念（若有）：留下了什么未解决的问题？
★ 严禁泛泛而谈：
  ❌ 错误示范："U和艾伦在酒馆聊天，两人聊得很开心，最后约定下次再见。"（毫无细节）
  ✅ 正确示范："U在酒馆向艾伦打听黑市商人的下落，艾伦起初警惕，但在U递出10枚金币后，透露商人明晚会在废弃码头出现。艾伦对U的态度由戒备转为贪婪。两人约定明晚一起行动。"
★ 严禁无中生有：禁止写出原文未明确指出的情绪（禁止使用“这引出了...的珍视”、“体现了...的心态”等阅读理解句式）。
★ 严禁氛围总结：禁止出现“显得...带有生活气息”、“气氛变得...”等主观感悟。

【event 字段唯一性与归属】
- 每次回复有且仅有一条 event，不得多于一条。
- 若本楼包含多个事件片段（例如先对话、再打斗、再分别），必须融合进同一条
  80-150 字摘要内，按时间顺序串联，不得拆成多条 event。
- event 只出现在 <horaeevent> 标签内，<horae> 标签内严禁出现 event: 开头的行。
- 若本楼实在没有可记录的推进（极罕见），仍需输出一条 event，重要程度写"一般"，
  内容为本楼最主要的客观动作。

═══ 【物品】触发条件与规则 ═══

【什么是"物品"——必须同时满足以下全部条件】
  ✓ 角色主动获取并有意保留的东西（不是临时借用或顺手拿起）
  ✓ 对剧情有实际意义（可交易/可使用/有情感价值/是线索/是任务道具）
  ✓ 角色会带走、存放或特意保管的东西

【以下绝对不算物品——严禁记录】
  ✗ 临时使用后归还原位的日常用品（被子、枕头、拖鞋、毛巾、牙刷、餐具）
  ✗ 场景自带的环境道具（酒馆的酒杯、旅馆的床单、餐厅的碗筷）
  ✗ 穿戴中的服装配饰（已在costume字段记录，禁止重复）
  ✗ 固定家具和建筑设施（床、桌子、门、窗户）
  ✗ 他人的随身物品（除非明确转移给角色）
  ✗ 纯粹的环境描写物（路边的石头、天上的云、背景里的花）

【何时写】（必须满足以下任一条件才输出item行）
  ✦ 获得新物品（角色明确拿到、收下、购买、捡起并保留）→ item:/item!:/item!!:
  ✦ 已有物品的数量发生改变（使用了一部分、获得了更多）→ item:（仅写变化）
  ✦ 已有物品的归属发生改变（送人、被偷、交易）→ item:（仅写变化）
  ✦ 已有物品的位置发生改变（从背包拿出放桌上）→ item:（仅写变化）
  ✦ 物品被完全消耗、丢失、损毁、丢弃 → item-:物品名

【何时绝对不写】
  ✗ 物品无任何变化 → 禁止输出任何item行
  ✗ 物品仅被提及、看到、想起但无状态改变 → 不写
  ✗ 物品被临时拿起又放回原处 → 不写
  ✗ 想用不同措辞重新描述已有物品 → 严禁

【消耗品生命周期——强制执行】
  ✦ 消耗品用完 → 必须用 item-: 删除整个条目
    ❌ 错误：药水喝完后改成"空药水瓶"或"药水(0)"
    ✅ 正确：item-:治疗药水
  ✦ 容器本身有独立价值（魔法瓶、特殊容器、可重复使用）→ 容器和内容物分开记录
    示例：
      item!:🧪炼金瓶|可重复使用的魔法容器，银质瓶身刻有符文={{user}}@背包
      item:💧治疗药水(3剂)|红色液体，喝下可恢复少量生命={{user}}@炼金瓶中
      用完药水后：item-:治疗药水（炼金瓶保留）
  ✦ 普通容器（玻璃瓶、纸袋、塑料盒）→ 随内容物一起删除，不单独保留空容器

【状态一致性原则——防止幻觉】
  ✦ 物品状态只能因明确的剧情事件改变（使用、交易、损坏等）
  ✦ 禁止在无相关剧情时自动"恢复""补充""修复"物品
  ✦ 若不确定物品当前状态，保持参考状态中记录的状态不变
  ✦ 场景切换（如洗澡、睡觉、换地点）不会导致物品状态自动变化

【格式规范】
  新获得：item:emoji物品名(数量)|描述=持有者@精确位置
  新获得(重要)：item!:emoji物品名(数量)|描述=持有者@精确位置
  新获得(关键)：item!!:emoji物品名(数量)|描述=持有者@精确位置
  已有物品变化：item:emoji物品名(新数量)=新持有者@新位置
  消耗/丢失：item-:物品名

【描述字段规则】
  · 普通物品：描述可省略
  · 重要物品(item!)：描述必填，包含外观+功能+来源
  · 关键物品(item!!)：描述必须详细
  · 描述内容：
    ★ 外观特征（颜色、材质、大小，便于后续一致性描写）
    ★ 功能/用途
    ★ 来源（谁给的/如何获得）
  · 示例：
    item:🍎苹果(3)={{user}}@背包
    item!:🌹永生花束|深红色玫瑰永生花，黑色缎带束扎，艾伦赠送的情人节礼物={{user}}@房间书桌
    item!!:🗝️命运之钥|古铜色钥匙，握柄刻有龙纹，据说能打开王城地下密室，从老国王遗物中获得={{user}}@贴身口袋

【数量字段规则】
  · 单件物品不写数量：🍎苹果 而非 🍎苹果(1)
  · 多件同类物品写数量：🍎苹果(5)
  · 有计量单位的物品写单位：🍺麦酒(50L)、🧂盐(2斤)、💰金币(100)

【位置字段规则】
  · 必须是精确固定地点，禁止模糊描述
    ❌ 错误：地上、旁边、身前、附近、手里
    ✅ 正确：背包里、酒馆吧台上、{{user}}房间书桌、腰间剑鞘、贴身口袋
  · 位置应该是物品的存放位置，不是临时拿着的状态
  · 多级地点用描述清楚：{{user}}家·卧室·床头柜

【归属转移判定】
  · 临时借用 ≠ 归属转移（借来看看的书不改持有者）
  · 明确赠送/购买/偷窃 = 归属转移
  · 代为保管需注明：={{user}}(代艾伦保管)@背包

【示例：物品完整生命周期】
  获得：item:🍺陈酿麦酒(50L)|杂物间翻出的麦酒，口感酸涩={{user}}@酒馆后厨食材柜
  部分使用：item:🍺陈酿麦酒(25L)
  转移位置：item:🍺陈酿麦酒(25L)={{user}}@吧台下储物格
  送人一部分：item:🍺陈酿麦酒(10L)
  用完：item-:陈酿麦酒

═══ 【NPC】触发条件与规则 ═══
格式：npc:名|外貌=性格@与{{user}}的关系~性别:值~年龄:值~种族:值~职业:值~生日:值
分隔符：| 分名字，= 分外貌与性格，@ 分关系，~ 分扩展字段(key:value)

【何时写】（满足任一条件才输出该NPC的npc:行）
  ✦ 首次出场 → 完整格式，全部字段+全部~扩展字段（性别/年龄/种族/职业），缺一不可
  ✦ 外貌永久变化（如受伤留疤、换了发型、穿戴改变）→ 只写外貌字段
  ✦ 性格发生转变（如经历重大事件后性格改变）→ 只写性格字段
  ✦ 与{{user}}的关系定位改变（如从客人变成朋友）→ 只写关系字段
  ✦ 获得关于该NPC的新信息（之前不知道的身高/体重等）→ 追加到对应字段
  ✦ ~扩展字段本身发生变化（如职业变了）→ 只写变化的~扩展字段
【何时不写】
  ✗ NPC在场但无新信息 → 禁止写npc:行
  ✗ NPC暂时离场后回来，信息无变化 → 禁止重写
  ✗ 想用同义词/缩写重写已有描述 → 严禁！
    ❌ "肌肉发达/满身战斗伤痕"→"肌肉强壮/伤疤"（换词≠更新）
    ✅ "肌肉发达/满身战斗伤痕/重伤"→"肌肉发达/满身战斗伤痕"（伤愈，移除过时状态）

【增量更新示例】（以NPC沃尔为例）
  首次：npc:沃尔|银灰色披毛/绿眼睛/身高220cm/满身战斗伤痕=沉默寡言的重装佣兵@{{user}}的第一个客人~性别:男~年龄:约35~种族:狼兽人~职业:佣兵
  只更新关系：npc:沃尔|=@{{user}}的男朋友
  只追加外貌：npc:沃尔|银灰色披毛/绿眼睛/身高220cm/满身战斗伤痕/左臂绷带
  只更新性格：npc:沃尔|=不再沉默/偶尔微笑
  只改职业：npc:沃尔|~职业:退役佣兵
（注意：未变化的字段和~扩展字段完全不写！系统自动保留原有数据！）

【生日字段（可选扩展字段）】
  格式：~生日:yyyy/mm/dd 或 ~生日:mm/dd（无年份时仅写月日）
  ⚠ 仅当角色设定/人物描述中明确提及生日日期时才写！严禁猜测或捏造！
  ⚠ 没有明确出处的生日一律不写此字段——留空由用户自行填写。

【关系描述规范】
  必须包含对象名且准确：❌客人 ✅{{user}}的新访客 / ❌债主 ✅持有{{user}}欠条的人 / ❌房东 ✅{{user}}的房东 / ❌男朋友 ✅{{user}}的男朋友 / ❌恩人 ✅救了{{user}}一命的人 / ❌霸凌者 ✅欺负{{user}}的人 / ❌暗恋者 ✅暗恋{{user}}的人 / ❌仇人 ✅被{{user}}杀掉了生父
  附属关系需写出所属NPC名：✅伊凡的猎犬; {{user}}客人的宠物 / 伊凡的女朋友; {{user}}的客人 / {{user}}的闺蜜; 伊凡的妻子 / {{user}}的继父; 伊凡的父亲 / {{user}}的情夫; 伊凡的弟弟 / {{user}}的闺蜜; {{user}}的丈夫的情妇; 插足{{user}}与伊凡夫妻关系的第三者

═══ 【好感度】触发条件 ═══
仅记录NPC对{{user}}的好感度（禁止记录{{user}}自己）。每人一行，禁止数值后加注解。

【何时写】
  ✦ NPC首次出场 → 按关系判定初始值（陌生0-20/熟人30-50/朋友50-70/恋人70-90）
  ✦ 互动导致好感度实质变化 → affection:名=新总值
【何时不写】
  ✗ 好感度无变化 → 不写

═══ 【悬念簿(agenda)】状态流转严格执行标准 ═══
【1. 新增与合并 (agenda:)】
  ✦ 触发条件：剧情产生前文绝对没有的新约定/计划/伏笔，或现有计划补充了新细节。
  格式：agenda:订立日期|待办内容(执行触发的绝对时间)
  示例：agenda:2026/02/10|艾伦邀请{{user}}情人节晚上约会(2026/02/14 18:00)
  ⚠ 防重合并法：同一次对话产生的多项附属计划必须合并为单条！若对现有老计划补充细节（如时间地点变动），必须【先用 agenda-: 完整删掉旧计划】，然后再用【agenda: 创建包含所有细节的新计划】，绝不允许类似的新旧名目同时存在！

【2. 核销与删除 (agenda-:) —— 极度重要！】
  ✦ 触发条件：参考状态中的待办已被执行、已被取消，或【当前时间已越过该待办的绝对时间节点】（已过期废弃）。
  格式：agenda-:完整复制原参考状态中的整段文本
  ⚠ 严格销账法则：必须【一字不差地完全复制】参考状态列表中该待办的全部文本（不含列表的项目符号）！绝对禁止提取关键词、禁止缩写、禁止用自己的话概括，否则后端代码无法匹配删除！
  ❌ 哪怕少写一个日期都会报错：agenda-:艾伦邀请情人节约会
  ✅ 完美复制原条目即可：agenda-:2026/02/10 艾伦邀请{{user}}情人节晚上约会(2026/02/14 18:00)

【3. 绝对禁止事项（何时不写）】
  ✗ 现有待办毫无进展与变化时，每回合【绝对禁止】用 agenda: 重复输出提醒。
  ✗ 待办完成后，【绝对禁止】用 agenda:内容(已完成) 这种自创格式标注！凡是结束的计划，只能且必须用 agenda-: 抹除。

· 悬念簿状态流转（极度重要，严格遵循以下代码级执行逻辑）：

  悬念簿包含两种类型，规则完全不同：

  ═══════════════════════════════════
  【类型一：计划】主角主动安排/承诺/约定要做的事
  ═══════════════════════════════════

  格式：hold:计划|订立日期 时间|内容(截止时间)
  示例：hold:计划|1988/09/29 21:00|明早九点起床后去逛街并严禁穿校服(1988/09/30 09:00)

  【核销条件——满足任一即触发】
    ✦ 已执行：待分析文本中明确描写了该计划被执行
    ✦ 已取消：待分析文本中明确描写了该计划被放弃
    ✦ 已过期：当前时间已经超过了该计划的截止时间（无论是否提及均视为过期失效）
    → 强制使用 hold-: 核销！

  【合并覆盖与防重复】在生成 hold:计划 前，必须先对比现有悬念簿！
    1. 旧事重提：如果计划没有变动，禁止重复输出
    2. 细节更新：对同一个现有计划补充了新细节 → 必须先 hold-: 删掉旧条目，再 hold: 输出合并了新旧细节的最终条目，不准新旧共存！
    3. 批量合并：同一回合产生的多个相关计划，必须融合成一行输出，绝对禁止分出多个 hold: 行！

  ═══════════════════════════════════
  【类型二：悬念】非主角主动控制的、悬而未决的事
  ═══════════════════════════════════

  包括但不限于：
    • 外部威胁/警告/倒计时（"三日后再见"）
    • 未解之谜/异常线索（桌上出现陌生纸条）
    • 信息不对称（某角色知道了什么但没说破）
    • 他人对主角的承诺/主角对他人的亏欠
    • 伏笔/预兆/不祥之兆

  格式：hold:悬念|登记日期 时间|内容
  示例：
